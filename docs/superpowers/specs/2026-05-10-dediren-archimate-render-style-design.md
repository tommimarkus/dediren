# Dediren ArchiMate Render Style Design

Date: 2026-05-10

## Purpose

Add an ArchiMate-oriented SVG render style next to the existing default and rich
SVG render policies, without creating a second styling system and without
making ELK layout depend on ArchiMate semantics.

The feature lets callers render a diagram with ArchiMate-like visual styling
when they already know the diagram uses ArchiMate semantic types.

## Decisions

- Keep one styling system: `svg-render-policy.schema.v1`.
- Add semantic type selectors to the existing SVG render policy.
- Add a separate render metadata artifact generated by projection.
- Keep render metadata style-free. It may contain semantic selector facts only.
- Keep `elk-layout` type-agnostic. ELK consumes layout intent and emits
  generated geometry, routes, groups, metrics, and warnings.
- Require `semantic_profile` matching between policy and render metadata before
  applying type selectors.
- Treat policy/metadata profile mismatch as a render error.
- Keep id overrides in the SVG render policy and make them win over type
  overrides.

## Contract Shape

Add a public render metadata contract:

```json
{
  "render_metadata_schema_version": "render-metadata.schema.v1",
  "semantic_profile": "archimate",
  "nodes": {
    "orders-component": {
      "type": "ApplicationComponent",
      "source_id": "orders-component"
    }
  },
  "edges": {
    "orders-realizes-service": {
      "type": "Realization",
      "source_id": "orders-realizes-service"
    }
  }
}
```

The keys under `nodes` and `edges` are layout-result node and edge ids. The
`source_id` fields preserve traceability back to the source graph but do not
carry presentation or layout data.

Render metadata must not contain colors, fonts, markers, shapes, line widths,
coordinates, spacing, or theme names. Those remain render-policy concerns.

Extend SVG render policy with optional semantic profile and type overrides:

```json
{
  "svg_render_policy_schema_version": "svg-render-policy.schema.v1",
  "semantic_profile": "archimate",
  "style": {
    "node_type_overrides": {
      "ApplicationComponent": {
        "fill": "#fff2cc",
        "stroke": "#7a5c00"
      }
    },
    "edge_type_overrides": {
      "Realization": {
        "stroke": "#374151",
        "stroke_width": 1.5
      }
    }
  }
}
```

## Data Flow

The layout and render-metadata paths are separate:

```text
source graph
  -> project --target layout-request
  -> layout request
  -> elk-layout
  -> layout result

source graph
  -> project --target render-metadata
  -> render metadata

layout result + render metadata + ArchiMate SVG policy
  -> svg-render
  -> styled SVG
```

Example commands:

```bash
dediren project \
  --plugin generic-graph \
  --target render-metadata \
  --view main \
  --input fixtures/source/valid-archimate-oef.json \
  > render-metadata.json
```

```bash
dediren render \
  --plugin svg-render \
  --policy fixtures/render-policy/archimate-svg.json \
  --metadata render-metadata.json \
  --input layout-result.json
```

## Ownership

`dediren-contracts` owns:

- `RENDER_METADATA_SCHEMA_VERSION`;
- `RenderMetadata`;
- render metadata node and edge selector structs;
- SVG render policy fields for `semantic_profile`, `node_type_overrides`, and
  `edge_type_overrides`.

`schemas/` owns:

- `render-metadata.schema.json`;
- updates to `svg-render-policy.schema.json`.

`generic-graph` owns:

- `project --target render-metadata`;
- mapping selected source node and relationship `type` values into render
  metadata for the selected view.

`dediren-core` and `dediren-cli` own:

- routing `--metadata` into render plugin requests;
- validating render metadata where render command envelopes require it.

`svg-render` owns:

- validating metadata/profile compatibility;
- resolving type-based styles;
- emitting structured diagnostics for missing metadata, profile mismatch,
  unknown metadata ids, or invalid policy/metadata combinations.

## Style Resolution

SVG render resolves styles in this order:

```text
base style
-> semantic type override from matching render metadata
-> id override
```

For example, an `ApplicationComponent` node receives the ArchiMate type style.
If `node_overrides.orders-component` also exists, the id override wins for that
specific node.

If a policy has no `semantic_profile` and no type overrides, existing rendering
continues to work without metadata. If a policy declares semantic type
overrides, metadata with the matching profile is required.

## Initial ArchiMate Policy

Add `fixtures/render-policy/archimate-svg.json` as a bundled example policy.
The first version should cover only the ArchiMate types present in the current
fixtures rather than claiming full notation coverage.

Expected initial coverage:

- nodes: `ApplicationComponent`, `ApplicationService`;
- edges: `Realization`;
- any additional exact type values already used by the chosen ArchiMate render
  fixture.

## Testing

Contract tests should verify:

- valid render metadata passes schema validation;
- unknown render metadata fields are rejected;
- SVG policy type override fields validate;
- policy/profile mismatch produces a render error.

Plugin and CLI tests should verify:

- `generic-graph` projects render metadata for selected view nodes and
  relationships;
- `svg-render` applies node and edge type overrides;
- id overrides take precedence over type overrides;
- render fails when a type-aware policy is used without metadata;
- render fails when policy and metadata profiles differ;
- `dediren render --metadata` passes metadata to the SVG render plugin;
- README examples show the ArchiMate render policy flow.
